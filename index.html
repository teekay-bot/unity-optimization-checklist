✓

✓

✓

Project Settings → Quality → Rendering → VSync Count → Don't Sync

✓

✓

Player → Publishing Settings → Compression Method
✓
Target Architectures: Only include needed
Player → Other Settings → Configuration → Target Architectures (Android) or Architecture (iOS). Exclude x86 on Android.

Mobile HDR is Off across all tiers. HDR on mobile doubles the color buffer bandwidth (from RGBA8 to RGBA16F), which is extremely costly on tile-based GPUs. Only enable if you absolutely need HDR post-processing, and profile carefully.

Draw Call Batching — Critical Conflicts

🚫
SRP Batcher and GPU Instancing are MUTUALLY EXCLUSIVE.
The SRP Batcher takes priority. If a GameObject's shader is SRP-Batcher-compatible, GPU Instancing is silently ignored — Unity will not instance it, even if "Enable GPU Instancing" is checked on the material. They cannot work together on the same object.
🚫
Dynamic Batching also conflicts with SRP Batcher.
In URP, Dynamic Batching only activates for objects that are NOT SRP-Batcher-compatible. Since most URP shaders are SRP-compatible by default, enabling "Dynamic Batching" in the URP Asset usually does nothing visible.

Batching Priority Order in URP

SRP Batcher → ✓ DEFAULT for all URP shaders | Highest priority. Reduces render-state changes.
Static Batching → ✓ Works WITH SRP Batcher | Complementary — merges static meshes.
GPU Instancing → ✗ Blocked by SRP Batcher | Requires breaking SRP compatibility or using DrawMeshInstanced.
Dynamic Batching → ✗ Blocked by SRP Batcher | Only runs on SRP-incompatible objects (<300 verts).
💡
Recommended batching strategy:
• Keep SRP Batcher ON globally — it's the biggest CPU-side win for most scenes.
• Use Static Batching for non-moving geometry (works alongside SRP Batcher).
• For many identical meshes (grass, foliage, bullets), use Graphics.DrawMeshInstanced or RenderMeshInstanced which bypass the SRP Batcher and send real GPU-instanced draw calls.
• Only break SRP compatibility per-material (via MaterialPropertyBlock or shader modification) when you've profiled and confirmed GPU Instancing is faster for that specific case.
• Dynamic Batching is rarely useful in URP — leave it Off unless you have many tiny SRP-incompatible meshes.
⚠
MaterialPropertyBlock breaks SRP Batcher. Setting per-instance data via Renderer.SetPropertyBlock() makes that renderer SRP-incompatible. If you need per-instance visual variation, prefer creating separate Material instances (SRP Batcher handles many materials with the same shader efficiently) or use GPU Instancing APIs explicitly.

Corrected Batching Checklist

✓

Material Inspector → Enable GPU Instancing. Only works on SRP-incompatible shaders. For mass-instancing, use Graphics.DrawMeshInstanced / RenderMeshInstanced APIs.

Performance Budgets (Practical Values)

⚠
Frame budget ≠ 1000÷FPS. Unity's official mobile optimization guide recommends using only ~65% of the raw frame time to leave thermal headroom. A 30fps target gives 33ms mathematically, but the practical budget is ~22ms. The remaining ~11ms prevents the device from overheating and throttling. PC has active cooling so the full frame time is usable.

🚫
With SRP Batcher, "Batches" ≠ performance cost. The SRP Batcher doesn't reduce batch count — it reduces SetPass calls (render state changes) by caching material data on the GPU. You can have 500 batches with only 10 SetPass calls and get great performance. Always check SetPass calls in the Profiler → Rendering module, not just the Batches number. The Frame Debugger shows exactly where SRP batches break and why.
💡
Where these numbers come from:
• Frame budget (mobile): Unity's official blog recommends ~65% of raw frame time to prevent thermal throttling on mobile devices without active cooling.
• Draw calls: Practical guideline from multiple shipped mobile titles and Unity performance experts: <200 batches for mobile low/mid-spec hardware, <2000 for desktop.
• Triangle counts: 100K–500K per frame for mid-range mobile (varies heavily with shader complexity and fill rate).
• Memory: Mobile texture budgets from shipped Unity AR/VR and game projects. Total app memory accounts for textures + meshes + audio + scripts + GC heap. Exceeding OS limits causes silent OOM kills.
⚠
Mobile thermal throttling is real. Even if your game hits 60fps in the first minute, sustained load causes the SoC to throttle after 3–5 minutes. Always profile in long sessions (10+ minutes) on target hardware. Aim for 30fps on mobile to leave ~35% GPU idle time as thermal headroom.

Asset Optimization

Apply these per-asset rules as you import content throughout development.

Textures

✓
Compression → ASTC (modern mobile) / DXT-BC (PC)
Select texture → Inspector → Platform Override tab → Format. PVRTC only for very old iOS.
✓
Disable Read/Write on textures not modified at runtime
Select texture → Inspector → Advanced → Read/Write (uncheck)
✓
Set Max Size to smallest acceptable resolution per texture
Select texture → Inspector → Platform Override tab → Max Size
✓
Disable Generate Mip Maps on UI / 2D sprites
Select texture → Inspector → Advanced → Generate Mip Maps (uncheck)
✓
Combine UI sprites into Sprite Atlases
Create: Assets → Create → 2D → Sprite Atlas. Then drag sprites into the Objects list.

Meshes & Rendering

✓
Use LODGroup on 3D meshes — simpler models at distance
Select GameObject → Add Component → LOD Group. Assign LOD meshes to each level.
✓
Disable LOD Crossfade on Mobile Low (use instant switch)
✓
Use Maximum LOD Level in Quality to strip unneeded LODs from build
✓
Minimize overdraw — avoid full-screen transparent overlays
✓
Bake Occlusion Culling for scenes with large static geometry
Window → Rendering → Occlusion Culling. Mark occluders/occludees as Static, then Bake.

Lighting

✓
Bake static lighting; use Mixed lighting mode
Window → Rendering → Lighting → Lightmapping Settings → Generate Lighting. Set lights to Mixed or Baked.
✓
Use Light Probes for dynamic objects
✓
Keep real-time lights minimal (ideally 1 Main Directional)
✓
Disable Realtime GI on mobile
Window → Rendering → Lighting → Realtime Global Illumination (uncheck)

Shaders

✓
Strip unused shader variants via Built-in Shader Stripping
✓
In URP Asset, disable features you don't use (Ambient Occlusion, Decals)
URP Asset → Lighting / Rendering sections. Each disabled feature strips its shader variants from the build.
✓
Use ShaderVariantCollection + Strict mode to identify unnecessary variants
Project Settings → Graphics → Shader Loading → Log Shader Compilation, plus Assets → Create → Shader → Shader Variant Collection

Audio & Particles

✓
Use compressed audio (MP3/OGG) on mobile; limit Max Real Voices
✓
Limit particle count/spawn rates; use Billboard shaders
✓
Disable Soft Particles and Motion Vectors on mobile

Pre-Release Build Checklist

Run through these before every release build.

✓
Development Build: Off
✓
Debug logging: Stripped ([Conditional] attribute or #if directives)
✓
Managed Debug Symbols: Stripped
✓
Profiler CPU/GPU: Frame times within budget (no large GC spikes)
✓
Draw Calls + SetPass Calls: Within tier budget (check Profiler → Rendering)
✓
Memory: Texture and asset memory under budget (no leaks)
✓
Resolution: Runs at intended resolution on all target devices
✓
SRP Batcher: Confirmed Enabled in Frame Debugger
✓
Depth/Opaque Textures: Off unless actively used by a shader/effect
✓
Quality Tier: Correct tier active on each device class
✓
Lightmapping: All static lights baked; real-time light count as planned
✓
Shaders: No compile errors, no fallback shaders, minimal variant count
✓
Addressables/IO: Asset loading is async and non-blocking
✓
Cameras: Only necessary cameras active; UI is batched
✓
Physics: Timestep and solver iterations match design; stable performance

Profiling & Device Testing

Profile early, profile often. Always test on real hardware.

Profiling

✓
Use Unity Profiler (CPU, GPU, Memory modules)
Window → Analysis → Profiler
✓
Check main thread vs. workers vs. render thread for CPU bottlenecks
✓
Use Frame Debugger to inspect SRP batches, SetPass calls, and batch-break reasons
Window → Analysis → Frame Debugger. Expand RenderLoopNewBatcher → Draw list to see per-batch info.
✓
Monitor GC pressure (allocations per frame)
Profiler → CPU module → Hierarchy view → GC.Alloc column. Target: 0 bytes in gameplay loops.
✓
Check Overdraw for GPU fill-rate bottlenecks
Scene View → top-left dropdown → Overdraw. Saturated colors = high overdraw.

Device Testing

✓
Test on actual hardware for each target tier (including lowest-spec device)
✓
Run long sessions to check for thermal throttling
✓
Use Development Build + Profiler for on-device profiling
✓
Test low-memory scenarios for OOM crashes
✓
Mobile: Use platform profiling tools (Perfetto, Xcode Instruments)

Editor Validation

✓
Test at target resolutions in Game view
Game view → Resolution dropdown → Add custom resolutions matching target devices
✓
Use Frame Debugger to find unbatched objects
Window → Analysis → Frame Debugger → look for individual Draw Mesh entries outside SRP Batch groups
✓
Verify SRP Batcher is active on all expected shaders
Select any shader in Project → Inspector shows "SRP Batcher: Compatible/Not compatible"
